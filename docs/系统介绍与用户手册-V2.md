# 第一部分：项目概述

## 1.1 研究背景与问题动机

### 1.1.1 通信系统的"固化之困"

当今 4G/5G 移动通信系统虽然功能强大，但它们有一个共同特点：**通信协议栈由人类工程师一次性设计并标准化完成，部署后难以做出结构性调整**。这带来了几个深层问题：

1. **场景适应性不足**：一套协议栈需要覆盖从城市宏蜂窝到农村广覆盖、从高速铁路到室内热点等千差万别的场景。在极端条件下（如灾害应急通信、卫星通信、无人集群），既定参数往往不是最优的。

2. **升级迭代周期漫长**：从 3GPP 标准化到设备商用通常需要数年时间。当新场景、新需求出现时，系统无法快速响应。

3. **"一刀切"的设计**：现有系统对所有用户、所有场景使用固定的调制编码表、固定的 HARQ 策略、固定的导频结构。而在动态环境中，这些"最优配置"可能瞬间变为"最差配置"。

### 1.1.2 AI 进入 RAN 的现状与局限

当前学术界和产业界（包括 3GPP Rel-18/19、O-RAN Alliance）正在探索 AI-RAN，主要做法是：

```text
传统 RAN 架构 + AI 局部优化
```

例如用 AI 增强信道估计、波束管理、资源调度、流量预测和节能控制。这些工作有价值，但本质上仍是 **"人工设计通信系统 + AI 局部优化"** 的范式——通信系统的基本架构、协议栈结构、模块边界仍然是人工预设的。

### 1.1.3 本项目试图回答的问题

> **能否进一步推进到"AI 按需生成通信能力"的新阶段？**

即：系统初始不预置完整、固定的空口和协议栈，而是保留一个**可信、稳定、低复杂度、可回退的最小通信内核**，在此基础上由 AI 根据任务需求、信道状态、频谱环境、终端能力、算力资源和 QoS 约束，动态选择、组合、配置甚至受限生成通信功能模块。

## 1.2 项目定位与核心创新

### 1.2.1 系统定位

**基于微内核的自演进 RAN 原型 (Microkernel-based Self-Evolving RAN Prototype)** 定位为面向未来 AI-Native RAN 的科研原型平台，是对当前 AI-RAN 范式的一次纵深推进。

核心思想可概括为：

| 维度   | 传统 RAN    | 本系统            |
| ---- | --------- | -------------- |
| 通信能力 | 人一次性固化设计  | **AI 按需动态生成**  |
| 协议栈  | 固定分层，预先实现 | **可插拔模块，动态组装** |
| 安全兜底 | 依赖完整协议栈   | **微内核作为保底生命线** |

### 1.2.2 核心创新点

1. **通信微内核**：借鉴 Linux 微内核思想，保留最小、可信、稳定的通信能力作为系统底座（低速可靠控制信道 + 基础同步 + 最小信令 + 回退机制），而非依赖完整协议栈。

2. **AI 受约束生成**：AI 不是在无边界空间内自由生成，而是在模块仓库、安全规则引擎和数字孪生验证的多重约束下进行"受约束创作"。

3. **全链路闭环验证**：所有 AI 生成的候选方案必须先进入数字孪生沙箱完成完整验证（接口检查 + 约束验证 + 链路仿真 + 故障注入 + 安全回退测试），只有综合评分合格的方案才能签名准入。

4. **安全自愈回退**：Runtime 中实时监控模块运行状态，一旦异常（BLER 超标、时延过限、模块崩溃），自动卸载异常模块并切回微内核保底通道。

5. **持续演进闭环**：运行数据反哺数字孪生模型校准 → 模块性能迭代 → AI 策略再生成 → 模块库版本更新，形成自学习、自优化、自演进闭环。

## 1.3 与传统 AI-RAN 范式的区别

```text
传统 AI-RAN：
  人工设计完整协议栈 → 部署上线 → AI 在固定框架内局部优化
  （如 AI 增强信道估计、AI 辅助调度）

本系统：
  最小可信微内核 → AI 按需生成/组合通信模块 → 数字孪生验证 → Runtime 动态部署 → 异常回退微内核
  （通信能力本身由 AI 生成，不止是优化）

| 对比维度       | 传统 AI-RAN | 本系统            |
| ---------- | --------- | ------------------ |
| **通信能力来源** | 人工预先设计    | AI 动态生成 + 组合       |
| **协议栈形态**  | 固化的分层实现   | 可插拔模块生态            |
| **AI 的角色** | 优化器（参数调优） | 生成器（模块配置/组合/受限代码）  |
| **安全机制**   | 依赖完整协议栈   | 微内核兜底 + 安全回退       |
| **适应周期**   | 月/年级标准化   | 秒/分级在线调整           |
| **适用场景**   | 常规蜂窝通信    | 应急/卫星/专用/无人集群等极端场景 |

## 2.2 核心设计原则

| 原则           | 说明                          | 来源          |
| ------------ | --------------------------- | ----------- |
| **微内核极简**    | 内核只保留生命线能力，不承载业务逻辑          | Linux 微内核思想 |
| **ABI 稳定**   | 模块接口主版本稳定，允许扩展不破坏兼容         | 操作系统设计      |
| **AI 受约束生成** | AI 在模块仓库、安全规则、数字孪生三重约束下创作   | Linux eBPF  |
| **先验证后部署**   | 所有方案必须经过数字孪生验证、评分才能上线       | 安全关键系统      |
| **控制面保底**    | 独立低速可靠控制信道始终存在，作为系统生命线和回退通道 | 电信级可靠性      |
| **可回退优先**    | 任何新能力失败时，系统自动卸载异常模块并切回微内核   | 故障安全设计      |
| **数据闭环**     | 运行数据反哺孪生模型和 AI 编排器，实现持续演进   | 持续学习        |

## 3.2 AI 智能体编排层 (`uran/ai/`)

> **功能**：接收自然语言任务需求，通过多智能体协同生成通信方案。
> **对应模块**：模块2

| 文件                          | 说明                                                                                   |
| --------------------------- | ------------------------------------------------------------------------------------ |
| `__init__.py`               | 模块导出接口，导出 AIOrchestrator、IntentSpec、PlanSpec 等                                       |
| `intent_schema.py`          | **意图与环境数据结构** — 定义 `IntentSpec`（用户需求规格）和 `EnvironmentSpec`（环境参数规格），将自然语言需求结构化为可计算的数据 |
| `plan_schema.py`            | **通信方案数据结构** — 定义 `PlanSpec`，包含调制方式、编码方案、编码率、MCS、HARQ配置、导频密度、发射功率、调度策略、功控策略等全部通信参数   |
| `agent_base.py`             | **智能体基类** — 所有 AI Agent 的抽象基类，定义 `process()` 统一接口和通用日志追踪能力                           |
| `intent_agent.py`           | **意图解析智能体** — 将中文自然语言需求（如"在高铁场景下需要高可靠性通信，BLER<1%"）解析为结构化的 IntentSpec（场景类型、优先级、性能目标等） |
| `environment_agent.py`      | **环境分析智能体** — 分析环境参数（SNR、移动速度、干扰水平、节点数量等），生成 EnvironmentSpec，支持外部传入环境参数              |
| `module_selection_agent.py` | **模块选择智能体** — 根据意图和环境，从通信模块库中选择合适的调制、编码、HARQ、调度等模块组合                                 |
| `parameter_gen_agent.py`    | **参数生成智能体** — 为选中的模块填充具体参数值（MCS等级、编码率、重传次数、发射功率等）                                    |
| `critic_agent.py`           | **方案审查智能体** — 对生成方案进行审查，标记潜在问题并给出改进建议                                                |
| `llm_adapter.py`            | **LLM 适配器（预留）** — 为未来接入大语言模型（GPT/Claude/本地模型）预留接口，当前使用规则基线                           |
| `orchestrator.py`           | **AI 编排器** — 多智能体协同总入口，协调五个 Agent 的顺序执行，生成多套候选方案                                     |

## 3.3 数字孪生验证层 (`uran/twin/`)

> **功能**：在虚拟沙箱中对候选通信方案进行五步验证，输出评分报告。
> **对应模块**：模块3

| 文件                   | 说明                                                                                                               |
| -------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `__init__.py`        | 模块导出接口                                                                                                           |
| `sandbox.py`         | **数字孪生沙箱** — 集成验证主入口，依次执行接口检查→约束验证→链路仿真→故障注入→综合评分，返回 `TwinValidationReport`，决策为 ACCEPT/NEED_REVISE/REJECT        |
| `validators.py`      | **接口与约束验证器** — `InterfaceValidator` 检查方案字段完整性（10个必需字段），`ConstraintValidator` 检查参数合法性（调制方式可支持性、编码率范围、MCS范围、功率上限等） |
| `link_simulator.py`  | **链路级仿真器** — `LinkLevelSimulator` 基于有效 SINR 计算 BLER/吞吐量/时延/频谱效率/能耗，考虑编码增益、导频效率、HARQ 增益、调制惩罚、干扰惩罚                 |
| `fault_injection.py` | **故障注入器** — `FaultInjector` 模拟 SNR 骤降、突发干扰、高移动性、功率限制、模块故障五种故障，测试方案的鲁棒性                                           |
| `scoring.py`         | **综合评分器** — `TwinScorer` 按优先级（可靠性/吞吐/时延/能耗/均衡/抗干扰）加权计算 0-100 分，包含鲁棒性分项                                           |
| `report.py`          | **验证报告数据结构** — `TwinValidationReport` 包含决策、各检查项通过状态、KPI 值、警告/错误/建议                                               |

## 3.4 通信微内核运行时 (`uran/runtime/`)

> **功能**：可信底座，负责模块加载、运行监控和安全回退。
> **对应模块**：模块4

| 文件                 | 说明                                                                                                                                   |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| `__init__.py`      | 模块导出接口，导出 MicrokernelRuntime、RuntimeState                                                                                            |
| `microkernel.py`   | **微内核核心** — `MicrokernelRuntime` 提供 `load_plan()` 加载方案、`step()` 单步执行、`run()` 批量执行、`inject_failure()` 故障注入、`fallback()` 安全回退，是系统的执行引擎 |
| `state_store.py`   | **状态存储** — `RuntimeState` 数据结构，保存运行时ID、活跃方案ID、活跃模块列表、状态(RUNNING/FALLBACK)、步数、最近KPI、事件列表、回退原因                                         |
| `events.py`        | **事件总线** — `EventBus` 记录系统运行事件（PLAN_LOADED、MODULE_FAILURE_INJECTED、FALLBACK_ACTIVATED 等），支持时间戳查询                                     |
| `monitor.py`       | **运行时监控** — `RuntimeMonitor` 收集每一步的 KPI 数据，支持近期 BLER 统计和综合摘要                                                                         |
| `safety_guard.py`  | **安全守护** — `SafetyGuard` 检测异常（模块健康检查失败、连续高 BLER），判定是否需要触发回退                                                                          |
| `module_loader.py` | **模块加载映射** — `map_plan_to_module_ids()` 将通信方案转换为可执行模块 ID 列表（如 QPSK→mod_qpsk、卷积编码→code_conv_1_2）                                      |

## 5.5 KPI 指标解读

| 指标                      | 全称                                      | 通俗解释                | 好的值       |
| ----------------------- | --------------------------------------- | ------------------- | --------- |
| **SNR**                 | Signal-to-Noise Ratio                   | 信号"清晰度"，越大信号越清晰     | >10dB     |
| **SINR**                | Signal-to-Interference-plus-Noise Ratio | 考虑干扰后的信号清晰度         | >8dB      |
| **BLER**                | Block Error Rate                        | 数据传错的概率，越小越好        | <5%       |
| **BER**                 | Bit Error Rate                          | 比特传错的概率，比BLER更细粒度   | <1%       |
| **Throughput**          | Throughput (kbps)                       | 每秒成功传输的数据量          | >500kbps  |
| **Latency**             | Latency (ms)                            | 数据从发到收的时间           | <50ms     |
| **Spectral Efficiency** | Spectral Efficiency (bps/Hz)            | 单位频谱带宽上的吞吐量         | >1 bps/Hz |
| **MCS**                 | Modulation and Coding Scheme            | "通信档位"，值越大速度越快但越易出错 | 0-28      |
| **EVM**                 | Error Vector Magnitude (%)              | 发射信号质量，越小越好         | <10%      |
| **Packet Loss**         | Packet Loss Rate                        | 网络丢包比例              | <1%       |

## 5.9 常见问题 FAQ

**Q1: 启动 Streamlit 时报 `streamlit: command not found`？**

A: 使用 `python -m streamlit run uran/dashboard/research_app.py`。

**Q2: 页面加载后没有数据？**

A: 需先点击对应页面的操作按钮。模块1 需先在上方控制栏点击「初始化」再点击「全部」；模块2–6 需点击「生成方案」/「启动演进闭环」等。

**Q3: 如何更换端口？**

A: 添加 `--server.port 8080` 参数。

**Q4: 能否在无 GUI 的服务器上运行？**

A: 可以。使用 `--server.headless true` 参数，通过浏览器远程访问。

**Q5: 仿真结果是否基于真实物理模型？**

A: 使用简化但趋势合理的模型。目标是演示全流程逻辑和可视化效果，而非严格复现 3GPP 物理层。

**Q6: 模块2 的 AI 是否接入了大语言模型？**

A: 当前使用规则基线（确定性决策），确保可复现性。`llm_adapter.py` 预留了 LLM 接入接口。

**Q7: 端到端流程的执行时间是多久？**

A: 纯计算时间 < 2 秒。包含 50 步 Runtime 执行 + SDR 模拟 + 演进分析。

**Q8: 如何自定义通信场景？**

A: 在 `configs/research_scenarios/` 下创建新的 YAML 配置文件，参照已有场景格式。然后在 Dashboard 的 SCENARIO_FILES 字典中添加映射。
