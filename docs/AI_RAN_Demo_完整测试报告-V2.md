## 一、系统简介：被测系统是什么？

### 1.1 通俗理解

本项目定位为面向未来 AI-Native RAN 的**基于微内核的自演进 RAN 原型 (Microkernel-based Self-Evolving RAN Prototype)**，是对当前 AI-RAN 范式的一次纵深推进。系统初始不预置完整、固定的空口和协议栈，而是保留一个**可信、稳定、低复杂度、可回退的最小通信内核**，在此基础上由 AI 根据任务需求、信道状态、频谱环境、终端能力、算力资源和 QoS 约束，动态选择、组合、配置甚至受限生成通信功能模块。

### 1.2 系统的五层结构（一句话版）

| 层级       | 通俗解释             | 类比             |
| -------- | ---------------- | -------------- |
| **仿真核心** | 模拟信号变化和通信效果      | 赛车游戏的物理引擎      |
| **AI编排** | 理解需求，生成通信方案      | 自动挡汽车的变速箱      |
| **微内核**  | 加载方案、运行、出错回退     | 手机操作系统的安全模式    |
| **数字孪生** | 在"虚拟世界"中验证方案是否可行 | 飞机制造前的风洞测试     |
| **SDR**  | 模拟真实无线电收发        | 模拟飞行的飞行模拟器     |
| **演进闭环** | 从经验中学习，越来越聪明     | AlphaGo 自我对弈进步 |
| **频段配置** | 不同频谱参数的灵活配置      | 汽车的手动/自动模式切换   |

### 1.3 测试做了什么？

我们用 **119 个自动化测试**逐一验证了各层中每个模块的行为：给它一个已知的输入，检查它是否输出了符合预期的结果。

## 三、链路自适应与仿真引擎（14 项测试）

> **通俗背景**：这组测试验证系统最底层的"物理引擎"——模拟信号强弱变化，以及系统如何根据信号好坏自动选择通信参数。

### 3.1 链路自适应决策（test_adaptation.py）

> **测试的是**：`LinkAdaptationController`，系统的"自动驾驶决策器"。
> **通俗理解**：就像自动挡汽车根据车速和路况自动换挡——信号好时用"高速挡"传更多数据，信号差时用"低速挡"确保不丢包。

---

#### 测试1：`test_high_snr_selects_high_mcs` — 信号好→选高速档

| 项目       | 内容                                             |
| -------- | ---------------------------------------------- |
| **测试目的** | 验证当信号质量很好时，系统会选择高速通信模式                         |
| **输入条件** | SNR=20dB（信号很强，类似离基站很近）                         |
| **预期行为** | MCS≥6（高档位），调制方式为16QAM或64QAM，编码率≥0.66           |
| **实际结果** | ✅ 通过                                           |
| **通俗解释** | 系统面对强信号时不会"浪费"——它会自动采用更高效的传输方式，让你在信号好的时候享受高速上网 |

> **知识点**：SNR=20dB 就像在安静的图书馆里听人说话——听得非常清楚。MCS（Modulation and Coding Scheme）可以理解为"通信档位"，越大越快。

---

#### 测试2：`test_low_snr_selects_robust_mode` — 信号差→选保底模式

| 项目       | 内容                                                |
| -------- | ------------------------------------------------- |
| **测试目的** | 验证当信号质量很差时，系统会切换到最可靠的保底模式                         |
| **输入条件** | SNR=2dB（信号很弱，类似在地下室或电梯里）                          |
| **预期行为** | MCS≤1（最低档位），调制为BPSK（最简单最稳定的方式），编码率≤0.33，HARQ重传≥3次 |
| **实际结果** | ✅ 通过                                              |
| **通俗解释** | 系统在信号差时自动降低期望——宁可传得慢一点，也要确保数据能正确送达。就像雾天开车减速慢行     |

> **知识点**：BPSK 是所有调制方式中最简单的一种——只用两个信号状态表示 0 和 1，就像用亮和灭两种灯光发信号，容错能力最强。

---

#### 测试3：`test_previous_high_bler_penalizes_decision` — 之前的错误率太高→自动降档

| 项目       | 内容                                                            |
| -------- | ------------------------------------------------------------- |
| **测试目的** | 验证系统有"记忆"——如果之前传输质量很差，会把通信档位再降低一档                             |
| **输入条件** | 两步相同的 SNR=11dB。第一步无历史BLER，第二步传入 history BLER=0.5（50%数据包出错！）   |
| **预期行为** | 第二步的 MCS ≤ 第一步的 MCS（系统降档了）                                    |
| **实际结果** | ✅ 通过                                                          |
| **通俗解释** | 就像你发现这条路虽然是高速公路，但最近总堵车——即使路况表面还行，你也会选择开慢一点以免出事故。系统对历史信息有"警惕心" |

---

#### 测试4：`test_medium_snr_selects_qpsk` — 中等信号→选中档模式

| 项目       | 内容                                |
| -------- | --------------------------------- |
| **测试目的** | 验证中等信号下选择折中方案                     |
| **输入条件** | SNR=7dB（不好不坏的中间状态）                |
| **预期行为** | 调制为 QPSK，HARQ 重传=2 次              |
| **实际结果** | ✅ 通过                              |
| **通俗解释** | 信号不好不坏时，系统既不激进也不保守——选一个保险但也不太慢的方案 |

---

### 3.2 信道时间线模拟（test_channel_timeline.py）

> **测试的是**：`ChannelTimeline`，系统的"天气模拟器"——模拟信号随时间如何变化。
> **通俗理解**：信道就是信号传播的"道路"。这条道路可能是平坦的（constant），可能是逐渐变差的（linear_fading），也可能突然出现障碍物（blockage）或干扰（jamming）。

---

#### 测试5：`test_linear_fading_decreases` — 线性衰减：信号逐渐变弱

| 项目       | 内容                           |
| -------- | ---------------------------- |
| **测试目的** | 验证"信号逐渐衰减"模式下，开始时的信号强于结束时    |
| **输入条件** | 初始 SNR=20dB，终止 SNR=0dB，100 步 |
| **预期行为** | 第 0 步的 SNR > 第 100 步的 SNR    |
| **实际结果** | ✅ 通过                         |
| **通俗解释** | 模拟用户从基站附近走到远处——距离越远，信号越弱     |

---

#### 测试6：`test_constant_channel` — 恒定信道：信号不变

| 项目       | 内容                              |
| -------- | ------------------------------- |
| **测试目的** | 验证恒定模式下，所有时间点的信号值相等             |
| **输入条件** | mode=constant, initial_snr=10dB |
| **预期行为** | 第 0 步 = 第 50 步 = 第 100 步        |
| **实际结果** | ✅ 通过                            |
| **通俗解释** | 模拟用户站在原地不动的情况——信号始终如一           |

---

#### 测试7：`test_jamming_spike` — 干扰尖峰：信号突然恶化

| 项目       | 内容                                          |
| -------- | ------------------------------------------- |
| **测试目的** | 验证干扰模式下，信号从正常突然跌落到极低值                       |
| **输入条件** | mode=jamming_spike, 初始 SNR=15dB, 底部 SNR=2dB |
| **预期行为** | 第 10 步为 15dB（正常），第 50 步为 2dB（受干扰）           |
| **实际结果** | ✅ 通过                                        |
| **通俗解释** | 模拟突然出现一个干扰器——信号瞬间恶化。在军事或应急场景中，这种"信号骤降"是常见威胁 |

---

#### 测试8：`test_sinusoidal_oscillates` — 正弦波动：信号周期震荡

| 项目       | 内容                                       |
| -------- | ---------------------------------------- |
| **测试目的** | 验证正弦波动模式下，不同时刻的信号值不一样                    |
| **输入条件** | mode=sinusoidal, 初始 SNR=20dB, 终止 SNR=0dB |
| **预期行为** | 第 25 步的信号 ≠ 第 75 步的信号                    |
| **实际结果** | ✅ 通过                                     |
| **通俗解释** | 模拟信号周期性上下波动——像潮汐一样有规律地变化                 |

---

#### 测试9：`test_sinr_with_interference` — SINR 扣除干扰

| 项目       | 内容                                                                              |
| -------- | ------------------------------------------------------------------------------- |
| **测试目的** | 验证 SINR 正确扣除了干扰（SINR = SNR - 干扰损失）                                              |
| **输入条件** | SNR=12dB, 干扰=3dB                                                                |
| **预期行为** | SINR = 12 - 3 = 9dB                                                             |
| **实际结果** | ✅ 通过                                                                            |
| **通俗解释** | SNR 是信号的"音量"；SINR 是扣除背景噪音后的"有效音量"。就像在嘈杂餐厅里听人说话——对方声音大（SNR高）不等于能听清（如果干扰大，SINR就低） |
